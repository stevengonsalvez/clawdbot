### Commit 2: Config-Driven Message Handlers (42c647f2c)

**Purpose**: Enable config-based routing that triggers immediate agent execution when messages match conditions. Solves the problem of important messages (like bug reports) being queued and missed when cron jobs wake the agent.

**Configuration Schema** (`src/config/types.hooks.ts`):
```typescript
{
  hooks: {
    internal: {
      enabled: true,
      messageHandlers: [
        {
          // Match conditions (all must be true to trigger)
          channelId?: string;        // "whatsapp", "telegram", etc.
          conversationId?: string;   // Specific chat/conversation
          from?: string;            // Specific sender
          contentPattern?: string;  // Regex pattern
          contentContains?: string | string[]; // Keywords

          // Handler behavior
          priority?: "immediate" | "queue"; // Default: "immediate"
          execution?: "exclusive" | "parallel"; // Default: "exclusive"

          // Message transformation
          message?: {
            template?: string;      // Full message template with placeholders
            prefix?: string;        // Add prefix to original message
            suffix?: string;        // Add suffix to original message
          },

          // Execution config
          model?: string;          // Override model
          thinking?: "low" | "medium" | "high" | "deep"; // Thinking level
        }
      ]
    }
  }
}
```

**Match Conditions** (`src/hooks/message-handler-match.ts`):
- `channelId`: Match specific channel (e.g., "whatsapp", "telegram")
- `conversationId`: Match specific conversation/chat
- `from`: Match sender identifier
- `contentPattern`: Regex pattern matching (full RegExp syntax)
- `contentContains`: Keyword matching (string or array)
- All conditions must be true for handler to trigger

**Handler Execution** (`src/hooks/message-handler-run.ts`):
- Uses `runCronIsolatedAgentTurn` for agent execution
- Priority modes:
  - `immediate`: Bypass queue, execute right away
  - `queue`: Normal queue flow
- Execution modes:
  - `exclusive`: Handler only (skip normal message processing)
  - `parallel`: Handler + normal processing
- Message transformation with placeholders:
  - `{original}`: Original message content
  - `{from}`: Sender identifier
  - `{channelId}`: Channel ID
  - `{conversationId}`: Conversation ID

### Architecture

**Hook Types**:
1. **Fire-and-forget hooks** (message:received) - Observe events, don't block
2. **Message handlers** - Config-driven routing with agent execution
3. **Command hooks** - command:new, command:reset, command:stop
4. **Gateway hooks** - gateway:startup
5. **Agent hooks** - agent:bootstrap

**Event Flow**:
```
Message arrives
    ↓
Trigger message:received hook (fire-and-forget)
    ↓
Check message handlers (if configured)
    ↓ (if matched)
Execute handler with runCronIsolatedAgentTurn
    ↓ (if execution: "exclusive", stop here)
Route to agent or queue normally
```

## Known Issues & Limitations

### Current Limitations

1. **Fire-and-forget only**: message:received hooks cannot modify messages or block processing
2. **No filtering at hook registration**: All message:received hooks run for every message (filtering must be done in handler)
3. **Channel metadata varies**: Not all channels provide the same metadata fields
4. **Testing requires real gateway**: Unit tests can't fully simulate multi-channel behavior
5. **Message handlers run before allowlist checks**: Handlers execute for all messages, even if sender is not in allowlist (this is intentional for logging/analytics use cases)

### Future Enhancements

From docs (lines 316-324):
- `session:start`: When a new session begins
- `session:end`: When a session ends
- `agent:error`: When an agent encounters an error
- `message:sent`: When an outbound message is sent

Potential message handler improvements:
- Allowlist filtering in match conditions
- Time-based conditions (e.g., only trigger during business hours)
- Rate limiting per handler
- Handler chaining (multiple handlers for same message)
